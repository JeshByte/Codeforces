My first observation was that there couldn't be that many palindromic numbers for the given range of n. So I wrote a brute force code to check that - and found out that there are only 500 such numbers for the given n. Thus we can easily iterate through all the possible palindromic numbers. Once that is possible, this problem basically turns into the classic dp problem Coin Change 2 - just here instead of coin denominations we need to use all the palindromic numbers. But we can't run the dp for each test case seperately as it would end up taking too much time. So just precompute the dp table once before each test case till 4e4 and then just fetch the answers from it.
