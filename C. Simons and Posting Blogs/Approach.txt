This is a pure implementation problem. First if we look into the addition of each blog/array, we just reverse it and then add the first occurences of each element. So we will process our input accordingly to make further logic easier to implement. Then in each iteration, basically the lexicographically shortest array is added(can be found easily using the inbuilt sort function). Now next time onwards, when we lexicographically compare the arrays, we shouldnt cosider the already added elements as their position wont change anymore. An easy way to handle this is to create a new container removing these elements and using that for the next iteration. The time complexity of this code is - n*(L*logL) where n is the number of blogs/arrays and L is the summation of the length of each array/blog. As both n and L are <=3000, this would easily work within the given bounds.
